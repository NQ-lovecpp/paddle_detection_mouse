## 二、关键实验洞察

### 洞察 1：数据量对 YOLOv3 的决定性影响

这是本次实验**最显著的发现**：

```
YOLOv3 + 1/3 数据（1942张）：mAP = 44.4%
YOLOv3 + 全量数据（5828张）：mAP = 91.5%
差距：+47.1 个百分点（3× 数据量 → 2× mAP）
```

YOLOv3 使用 MobileNetV1 骨干，COCO 预训练权重的 head 需要从 80 类重新训练到 2 类，对训练数据的需求量大。1,942 张不足以让 head 层充分学习特定域（实验室场景）的特征，导致模型卡在局部最优（~44%）。全量 5,828 张则提供了足够的样本多样性。

**结论**：对于有监督目标检测，数据工程比模型选择更重要。

---

### 洞察 2：PicoDet-S 全面优于 YOLOv3（准确率、速度、体积三维胜出）

```
                    PicoDet-S L1    YOLOv3 Y4（最优）
最优 mAP@0.5       94.30%          91.54%     (+2.76%)
模型大小            ~4 MB           92 MB      (23× 更小)
推理 FPS (T4 GPU)  ~78 FPS         ~35 FPS    (2.2× 更快)
达到 91% 所需时间   ~30 epoch(90min) 70+ epoch(12h+)
```

关键原因：PicoDet-S 使用 LCNet 骨干 + LCPAN 颈部，专为移动端设计，COCO 预训练特征表示能力更强，anchor-free 设计对小数据集更友好（无需 anchor 聚类匹配）。

---

### 洞察 3：Batch Size 与精度的非直觉关系（PicoDet）

```
L1（bs=32,  lr=0.08）：94.30% ← 最优
L2（bs=64,  lr=0.16）：93.80%   ← 标准双卡配置
L3（bs=96,  lr=0.24）：94.10%
L4（bs=64,  lr=0.16）：93.50%（600e）
```

**反常现象 1**：单卡小 batch（L1）精度高于双卡标准 batch（L2），尽管 L1 训练时间是 L2 的 2.2 倍。
- 原因：小 batch 的梯度噪声对仅 5,828 张的小数据集起到了隐式正则化作用，防止过拟合。
- 量化：L1 每轮 182 步，L2 每轮 91 步（参数更新次数减半）。更多的梯度更新步数使模型探索更充分。

**反常现象 2**：L3（bs=96）比 L2（bs=64）更好，尽管 batch 更大。
- 原因：L3 的学习率 0.24 比 L2 的 0.16 更高，更大的学习率步长帮助跳出了 L2 所卡的局部最优。
- 这体现了 Linear Scaling Rule 在小数据集上的局限性——公式正确，但不同 LR 对应的优化轨迹有质的差别。

**反常现象 3**：L4（600e）比 L2（300e）更差（93.50% vs 93.80%），训练时间却是 2 倍。
- 原因：PicoDet-S 在约 70 epoch 达到峰值（94.30%），之后验证集 mAP 持续下滑（过拟合）。300 epoch 结束时已是 91.42%，600 epoch 延续了这一趋势（89.27% 结束），反而损失了 best_model 后期的微弱回升机会。

---

### 洞察 4：双卡 TCP 通信的实际效果

环境：双 Tesla T4，无 InfiniBand，使用 GLOO（TCP socket）fallback。

```
YOLOv3 Y3（单卡 17.4h） vs Y4（双卡 8.8h）：实际加速 1.97×，精度基本持平（+0.2%）
PicoDet L1（单卡 14.9h）vs L2（双卡 6.7h）：实际加速 2.22×，精度略降（-0.5%）
```

TCP 通信开销可接受，双卡加速比接近理论值 2×，说明对于小模型（PicoDet 4MB 参数同步量很小），通信开销可以忽略。

---

### 洞察 5：mAP 曲线形态差异揭示模型特性

**PicoDet-S**（L1 为例）：
```
epoch  10:  91.16%  ←  快速起跳（COCO预训练迁移效果立竿见影）
epoch  70:  94.30%  ←  峰值（约 3.5 小时训练）
epoch 100:  94.19%  ←  平台
epoch 150:  93.60%  ←  开始下滑
epoch 299:  91.42%  ←  回到接近起点（明显过拟合）
```

**YOLOv3 Y3**（全量数据）：
```
epoch   5:  81.80%  ←  起跳较低（head需要重新学习）
epoch  30:  87.13%  ←  缓慢上升
epoch  60:  90.03%  ←  仍在上升，尚未收敛
epoch  80:  91.30%  ←  终点，还没到峰值
```

YOLOv3 收敛慢、斜率平缓，需要更多 epoch 才能达到最优。如果 Y3/Y4 跑 120 epoch，可能突破 92%。

## 四、面试复述建议

### 核心叙事框架（STAR 法则）

**Situation（背景）**：
> "这是上海交通大学医学院网络信息中心的一个实验鼠行为分析项目。需要在双 Tesla T4 的服务器上，从零搭建 PaddleDetection 训练环境，完成一个二分类目标检测系统（mouse/other）并部署到 iOS 移动端。"

**Task（任务）**：
> "我负责全流程：整合三个异构数据源、标注清洗、设计对比实验、调优训练超参，最终导出 ONNX 模型并在 React Native + ONNX Runtime 实现 iOS 部署。"

**Action（行动）—— 重点展开）**：

可以围绕三个亮点展开：

**亮点 1：实验设计方法论**
> "我设计了两套 2×2 对比实验矩阵：PicoDet-S 的 `卡数 × batch size`（4 组），以及 YOLOv3 的 `数据量 × 卡数`（4 组），共 8 轮。学习率严格按 Linear Scaling Rule 推导：`lr = base_lr × (total_bs / reference_bs)`，保证控制变量。"

**亮点 2：意外发现和分析（这是面试亮点）**
> "在 PicoDet 实验里发现一个反直觉的结果：单卡小 batch（bs=32）精度 94.3%，反而比双卡标准 batch（bs=64）的 93.8% 更高。分析原因是：我们的训练集只有 5,828 张，小 batch 的梯度噪声对这个规模的数据集起了隐式正则化作用。另外更有意思的是，YOLOv3 用三分之一数据只能跑到 44% mAP，但用全量数据能达到 91.5%，+47 个百分点。这个结论强化了我对'数据是 ML 系统第一优先级'这个认知。"

**亮点 3：工程细节**
> "双卡通信上遇到了 NCCL 不可用的问题——服务器没有 InfiniBand，框架 fallback 到 GLOO（TCP socket）。实际测试双卡加速比约 2×，说明对我们的模型规模 TCP 通信开销可接受。"

**Result（结果）**：
> "最终 PicoDet-S 达到 94.3% mAP@0.5，比 YOLOv3 最优结果（91.5%）高 2.8 个百分点，模型只有 4MB（YOLOv3 是 92MB），推理速度约 78 FPS（YOLOv3 约 35 FPS）。将其导出为 ONNX 格式，在 React Native + ONNX Runtime 上完成了 iOS 部署雏形，推理管线打通。"

### 可能被追问的问题 & 参考回答

**Q: 为什么不用 YOLOv8/更新的模型？**
> "当时实习环境固定在 PaddleDetection 2.6 框架，配合单位已有的 Paddle 生态（Paddle Inference、Paddle Lite）。但架构上我们做了对比——PicoDet-S 是 2022 年 PaddlePaddle 专门为移动端优化的模型，性能已经很有竞争力。对于今天的部署需求，我评估了 YOLO26n（Ultralytics 2026 年 1 月发布），其 NMS-free 设计对 iOS CoreML 部署更友好，如果重做会考虑引入。"

**Q: 数据集怎么处理的？**
> "整合了三个来源：一是实验室自采的实验鼠图像（VOC XML 标注），二是开源数据集，三是网络爬取并人工清洗的样本。主要工作是标签统一（统一类别名和标注格式到 VOC）、去重、处理遮挡标注不规范的样本。最终 7,286 张，train/val = 5,828/1,458。"

**Q: 模型部署遇到了什么坑？**
> "最大的坑是 YOLOv3 导出的 ONNX 模型里内嵌了 `multiclass_nms3` 这个 PaddleDetection 自定义算子，CoreML 不认识，无法走 Metal GPU 加速，导致 iOS 上只有 1fps。切换到 PicoDet-S 之后，模型只有 4MB，输入从 608×608 降到 320×320，算子简单，CoreML 可以完整加速，预期 15-30fps。"

**Q: 为什么 L4（600 epoch）比 L1（300 epoch）差？**
> "因为这个数据集太小，模型在约 70 epoch 就达到了泛化能力峰值，之后验证集 mAP 开始持续下滑——这是经典的过拟合现象。延长训练只是让模型更深度地记住了训练集，best_model 机制虽然保存了峰值权重，但后续训练的 EMA 和 checkpoint 反而往更过拟合的方向走。理想做法是引入 early stopping 或增加 dropout，但对于 PicoDet-S 这个规模的模型在移动端部署场景，其实 70 epoch 足够。"

**Q: 学习率缩放的依据是什么？**
> "Linear Scaling Rule，来自 Facebook 2017 年的论文《Accurate, Large Minibatch SGD: Training ImageNet in 1 Hour》。原理是：当 batch size 扩大 k 倍时，每步梯度是 k 个样本的平均，信噪比提高；为了保持等效的参数更新幅度，学习率也乘以 k。具体在我们的 YOLOv3 实验里：官方参考值 lr=0.01 @ bs=64，双卡 bs=16 时 lr=0.01×(16/64)=0.0025。"
